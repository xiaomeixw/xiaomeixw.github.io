【范式10月】Android-MVP范式编程(一)

发表于 2015-10-03

我将保证MVP核心范式编程的文章将不少于5篇文章，在我还没有正式给MVP下定义前，我相信一个优秀的范例将非常有效的让我们对MVP有一个非常深入的了解。

首先让我们搭建一个基础MVP架构的目录，然后我们逐步解析MVP的含义，并最终完成一套符合The Clean Architecture核心思想的MVP架构封装。这里将是MVP范式编程的第一个章节,我将通过一个层级范例来解构MVP。

解析MVP层级范例

mvp的分层我们按照model-presenter-view三个包级别分层，这样的分层将非常方便于各层级的关系展示。

MODEL:

model包中装载的是我们通常所谓的java-bean对象，这些model对象继承Parcelable实现序列化。

PRESENTER:

presenter包中装载的是我们的逻辑处理对象，它将沟通view，并获取后台逻辑数据，将json解析为model对象，并根据数据请求的结果最终通知view层完成UI的更新.

VIEW:

view包中装载两个组成部分，一个是接口abstract-view，它将沟通activity-view和presenter，里面定义的都是等待activity-view实现的逻辑方法。  另一个是activity-view，它将完成view的所有实现。

MVP本质：

1.将view和data抽象分离处理，原activity中的两个组成部分：界面和数据逻辑分别放在view和presenter层级中。

2.view层主要是原activity的界面层级。它将分成两个组成部分，一个为activity-view(实际View业务处理)，另一个为abstract-view(定义抽象方法完成view层与presenter层沟通);

3.presenter层主要是原activity的数据逻辑层级。它将完成后台的数据逻辑请求，并将json转成model，通过请求的数据结果，将model返回给view层级，完成UI界面的更新。

1.model中的User.class将继承Parcelable实现序列化

package xiongwei.mvp.model;

import android.os.Parcel;
import android.os.Parcelable;

/**
 * Created by 伟 on 2015/10/3.
 * 实现Parcelable非常简单
 * 1）implements Parcelable
 * 2）重写writeToParcel方法
 * 3）重写describeContents方法，默认返回0即可
 * 4）实例化静态内部对象CREATOR实现接口Parcelable.Creator
 *
 * 在android studio中实现Parcelable的技巧非常简单
 * 1.先将你需要的的字段书写出来
 * 2.alt+Enter完成未完成方法(implements methods)的提示,此时AS会自动帮你完成describeContents和writeToParcel方法
 * 3.继续alt+Enter完成(Add Parcelable Implements)的提示,此时AS会完成实例化静态内部对象CREATOR。
 */
public class User implements Parcelable {

    private long id;
    private String name;
    private String url;
    private String email;
    private String login;
    private String location;
    private String avatarUrl;


    protected User(Parcel in) {
        id = in.readLong();
        name = in.readString();
        url = in.readString();
        email = in.readString();
        login = in.readString();
        location = in.readString();
        avatarUrl = in.readString();
    }

    public static final Creator<User> CREATOR = new Creator<User>() {
        @Override
        public User createFromParcel(Parcel in) {
            return new User(in);
        }

        @Override
        public User[] newArray(int size) {
            return new User[size];
        }
    };

    @Override
    public int describeContents() {
        return 0;
    }

    @Override
    public void writeToParcel(Parcel dest, int flags) {
        dest.writeLong(id);
        dest.writeString(name);
        dest.writeString(url);
        dest.writeString(email);
        dest.writeString(login);
        dest.writeString(location);
        dest.writeString(avatarUrl);
    }
}

public class BannerTime {

    /**
     * 初始化,其实就是最简单的创建BannerTime对象
     */
    public static BannerTime init(Context context) {
        if (sInstance == null) {
            sInstance = new BannerTime(context);
        }
        mContext = context;
        //初始化最初的一些参数
        init();
        //通过返回BannerTime对象,接着可以继续调用BannerTime中的其他方法
        return sInstance;
    }

    /**
     * 设置显示的时间参数
     */
    public static BannerTime shownTime(int time) {
        SHOW_TIME_BANNER = time;
        return sInstance;
    }

    /**
     * 放置图片
     */
    public static BannerTime imageBanner(String path, boolean crop) {
        requestCreator = Picasso.with(mContext).load(path);
        if (crop)
            requestCreator.fit().centerCrop();
        else
            requestCreator.fit().centerInside();
        return sInstance;
    }

    /**
     * 最终展示出来
     */
    public static void show() {
        //调用的还是带有回调的show(listener)
        show(null);
    }

    /**
     * 最终走实质性逻辑代码的地方,会将之前所有参数的部分进行处理
     * @param listener
     */
    public static void show(BannerTimeListener listener) {
        if (sInstance == null)
            throw new RuntimeException("It´ necessary call '.init()' method");
        mListener = listener;
        ImageView img = (ImageView) mView.findViewById(R.id.imageAd);
        if (requestCreator != null) {
            if (mPlaceHolder != 0) {
                requestCreator.placeholder(mPlaceHolder)
                        .into(img);
            } else {
                requestCreator.into(img);
            }
        } else {
            Picasso.with(mContext)
                    .load(mPlaceHolder)
                    .placeholder(mPlaceHolder)
                    .fit().centerCrop()
                    .into(img);
        }
        //创建dialog
        createDialog();
    }

}

//计时器方案关闭banner
scheduleHideBanner(dialog);

/**
 * 在指定的时间内关闭dialog
 */
private static void scheduleHideBanner(final Dialog dialog) {
    dismissDialog(dialog);
    //创建TimerTask
    timerTaskHide = new TimerTask() {
        @Override
        public void run() {
            scheduleShowBanner(dialog);
        }
    };
    timerHide.schedule(timerTaskHide, HIDE_TIME_BANNER);
}

 private static void dismissDialog(final Dialog dialog){
    //主线程运行
    ((Activity) mContext).runOnUiThread(new Runnable() {
        @Override
        public void run() {
            //一定要先判读activity是否关闭,否则会导致dialog无法附着在activity导致奔溃
            if (!((Activity) mContext).isFinishing() && dialog.isShowing()) {
                dialog.dismiss();
                if (mListener != null)
                    mListener.onDismiss();
            }
            Log.i(TAG, "hideBanner");
        }
    });
}

如何将你的Activity抽分为MVP架构：

在原生的处理业务逻辑中，我们会将"layout布局UI的显示"和"数据的获取"这两部分都放在一个Activity中，抽取为MVP架构的核心就在于将"数据获取"部分抽取出来放到presenter中，同时通过interface来完成与View(新Activity)的交互沟通。

MVP的几个核心点：

  1. View不直接与Model交互，View而是通过与Presenter交互来与Model间接交互： User<—>View<—>Presenter<—>Model
  2. 通常View与Presenter是一对一，但复杂的一个View可能会对应多个Presenter。

  3. 在android上践行MVP，在现有的解决方案上(2015年9月)必须或者需求性依赖一些第三方的lib来实现.比如：

    Retrofit(表述性状态传递–类型安全的 REST 客户端):实现标准的API接口解析与调用,以此简化API接口的调用和解析。

    Dagger(依赖注入):通过注解注入对象，从而完成自动的初始化，以此简化View的获取使用和处理。

    Rx(响应式编程):通过观察与订阅实现回调函数的监听和处理，以此减少android极度依赖回调函数处理逻辑的方案。
