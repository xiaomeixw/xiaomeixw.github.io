【正谈10月】Explore NoSql on Android

发表于 2015-10-23

一.前言

D.Richard Hipp不曾想到在这一年他萌芽的轻量级数据库居然诞生在一条漂泊在大海的军舰上，这个诞生于2000年8月的Sqlite数据库由13万行C代码构成,轻量简洁稳定的优势特点,使它经过多年的发展成为Google&Apple在移动端上探索数据库的核心主力，广泛而且大量的被使用。9年后,也就是2009年，NoSql形态数据库被逐渐推上历史的舞蹈核心，Mongodb在服务端架构数据库中大行其道，许多中小型数据库均开始由MySql转向更友好形态的Mongodb.而此时距离iphone1代发布已经过去了整整2年。
尽管如此，但在移动端领域NoSql形态数据库仍然还是处于萌芽起步阶段，在写这篇博客时我不经问自己:在Android上探索NoSql型数据库是否为时尚早?也许我们仍然需要时间来检验这个问题和它可能存在的标准答案！

二.何为Nosql

泛指非关系型的数据库，关系数据库关注在关系上，NoSQL关注在存储上。
NoSQL数据库种类繁多，但是一个共同的特点都是去掉了关系数据库的关系型特性。数据之间无关系，容易扩展。
当前主流四种主要形态的NoSql型数据库：

  • 键值（key-value）比如：Oracle BDB
  • 列存储数据库 比如：HBase
  • 文档型数据库 比如：MongoDb
  • 图形(Graph)数据库 比如：Infinite Graph

主流的Nosql型数据库在Android端的探索主要是以key-value键值对的形式存在。这一类数据库主要会使用到一个哈希表，这个表中有一个特定的键和一个指针指向特定的数据。Key/value模型对于IT系统来说的优势在于简单、易部署。但是如果DBA只对部分值进行查询或更新的时候，Key/value就显得效率低下了。

使用场景：

  • 1.取代SP存储；
  • 2.用于缓存场景.

三.探索NoSql

在这里我会探索一个简单的基于kryo序列化的Nosql，它的核心和JakeWharton的DiskLruCache有异曲同工之妙，最终一样也是会以File文件的形态存在。很多大型的APP，比如网易新闻等都是以DiskLruCache磁盘缓存为核心构筑自己的缓存策略的。今年10月份之前的美丽神器APP也是用的DiskLruCache来处理缓存的。在我将它置换为NoSql型数据后，速度相比较于DiskLruCache快了接近3倍以上。

3.1 何为kryo

Kryo 是一个快速高效的Java对象图形序列化框架，它原生支持java，且在java的序列化上甚至优于google著名的序列化框架protobuf。在插入一条数据时比原生java的序列化最快可以达到17倍之多。

2014年10月2日发布了3.0的版本.

项目地址:https://github.com/EsotericSoftware/kryo build byEsoteric Software.

API:

Kryo kryo = new Kryo();
// ...
Output output = new Output(new FileOutputStream("file.bin"));
SomeClass someObject = ...
kryo.writeObject(output, someObject);
output.close();
// ...
Input input = new Input(new FileInputStream("file.bin"));
SomeClass someObject = kryo.readObject(input, SomeClass.class);
input.close();

我们知道数据库离不开增删查改这4个最基本的逻辑方法，所以我们抽出一个接口：我们知道有时改其实也就是重新增，所以这里只定义4种方法:增 、查 、判断 和 删

    boolean isExist(String key);

    /**删:如果存在删除*/

}

@Override
    public <T> void save(String key, T value) {

        //第一次调用file肯定不存在,那就创建它
        createFile();

        //包装value内容
        final SqlTable<T> sqlTable = new SqlTable<>(value);

        //创建一个原始orgFile和BackupFile
        final  File oriFileDir = SqlHelper.getOriFileDir(mFilesDir, key);
        final File backFileDir = SqlHelper.makeBackupFile(oriFileDir);

        //将原文件重命名back文件
        SqlHelper.handlerOriginalFileToBackFile(oriFileDir, backFileDir);

        //写入文件
        writeTableFile(key, sqlTable, oriFileDir, backFileDir);

    }

 private <T> void writeTableFile(String key, SqlTable<T> sqlTable, File oriFileDir, File backFileDir) {
        FileOutputStream fileStream = new FileOutputStream(oriFileDir);
        final Output kryoOutput = new Output(fileStream);

        //操作kryo
        getKryo().writeObject(kryoOutput, sqlTable);
        kryoOutput.flush();
        fileStream.flush();
        SqlHelper.sync(fileStream);
        kryoOutput.close(); 

        //删除备份文件        
        backFileDir.delete();      
}

get:

/**
 * 根据key获取value
 * @param key
 * @param <T>
 * @return
 */
@Override
public <T> T get(String key) {
    //如果不存在创建文件夹
    createFile();

    //拿到两个File文件
    final File originalFile = SqlHelper.getOriFileDir(mFilesDir, key);
    final File backupFile = SqlHelper.makeBackupFile(originalFile);

    if (backupFile.exists()) {
        originalFile.delete();
        backupFile.renameTo(originalFile);
    }

    //key对应的Value是否存在
    if (!isExist(key)) {//如果没有那就返回null
        return null;
    }

    //kryo读取
    return readTableFile(key, originalFile);
}


private <T> T readTableFile(String key, File originalFile) {

     final Input i = new Input(new FileInputStream(originalFile));
     final Kryo kryo = getKryo();

      final SqlTable<T> paperTable = kryo.readObject(i, SqlTable.class);
      i.close();
      return paperTable.mContent;

}

    createFile();

    final File originalFile = SqlHelper.getOriFileDir(mFilesDir, key);
    if (!originalFile.exists()) {
        return;
    }

    boolean deleted = originalFile.delete();
    if (!deleted) {
        throw new SqlException("Couldn't delete file " + originalFile

3.5.3 NoSqlService

NoSqlService类:类似JavaEE中的Service层，主要是解耦

/**
 * Created by xiongwei,An Android project Engineer.
 * Date:2015-11-02  19:41
 * Base on Meilimei.com (PHP Service)
 * Describe: 解耦
 * Version:1.0
 * Open source
 */
public class NoSqlService {

    private final NoSqlDao mStorage;

    public NoSqlService(Context context, String dbName) {
        mStorage = new NoSqlDao(context.getApplicationContext(), dbName);
    }


    public void delete() {
        mStorage.delete();
    }


    public <T> NoSqlService write(String key, T value) {
        if (value == null) {
            throw new SqlException("noSql doesn't support writing null root values");
        } else {
            mStorage.save(key, value);
        }
        return this;
    }


    public <T> T read(String key) {
        return read(key, null);
    }


    public <T> T read(String key, T defaultValue) {
        T value = mStorage.get(key);
        return value == null ? defaultValue : value;
    }


    public boolean exist(String key) {
        return mStorage.isExist(key);
    }


    public void delete(String key) {
        mStorage.deleteIfExists(key);
    }
}

3.5.4 NoSqlManager

NoSqlManager主要做管理，同时异步线程开启get/save这样的耗时操作

首先单例模式(双重锁判断)：

public static NoSqlManager getDbManager(Context context) {
    return  getDbManager(context,null);
}

public static NoSqlManager getDbManager(Context context,String dbName) {
    if (dbCache == null) {
        synchronized (NoSqlManager.class) {
            if (dbCache == null) {
                dbCache = new NoSqlManager(context,dbName);
            }
        }
    }
    return dbCache;
}

构造:初始化service

private NoSqlManager(Context context, String dbName){
    this.mContext= context;
    this.mDbname=dbName;
    getNoSQL();
}

private static NoSqlService getNoSQL() {
    if (mContext == null) {
        throw new SqlException("context must init in application or activity");
    }

    if(mDbname==null){
        mDbname=Constans.DEFAULT_DB_NAME;
    }

    synchronized (mSqlMap) {
        noSQL = mSqlMap.get(mDbname);
        if (noSQL == null) {
            //初始化NoSqlService
            noSQL = new NoSqlService(mContext, mDbname);
            mSqlMap.put(mDbname, noSQL);
        }
        return noSQL;
    }
}

private  class  DbAsyncTask<T> extends AsyncTask<Object, Void, T> {


    public DbAsyncTask(String get_key,Callback callback) {
        this.get_key=get_key;
        this.callback=callback;
    }


    public  DbAsyncTask(String save_key,T buffer) {
        this.save_key=save_key;
        this.buffer=buffer;
    }

    @Override
    protected T doInBackground(Object... params) {
        switch ((Integer) params[0]) {
            case MESSAGE_SAVE_DB:
                //2.在后台操作中存储
                saveDbInternal(save_key, buffer);
                return null;
            case MESSAGE_GET_DB:
                return getDbInternal(get_key);
            case MESSAGE_DELETE_DB:
                return deleteDbInternal(delete_key);
            ....
        }
        return null;
    }

    @Override
    protected void onPostExecute(T result) {
        super.onPostExecute(result);
        if(result!=null){
            Log.d("onPostExecute",result+"");
            callback.onFinish(result);
        }
    }
}

//3.调用service层操作(service再调用NoSqlDao)
private <T> void saveDbInternal(String key, T value) {

3.6 配合缓存策略使用

  • QuickUtils.cache.has(key) 方法调用的是NoSqlManager.has(key)方法判断该key是否有value
  • QuickUtils.cache.getSynchronous(key); 方法调用的是NoSqlManager.get(key).
  • QuickUtils.cache.setSynchronous(key, value); 方法调用的是NoSqlManager.save(key, value).
@Override
protected void initData() {
    //判断是否有没有网络，同时缓存中确实存在该key
    if(!QuickUtils.net.hasNetwork() && QuickUtils.cache.has(key)){
        //有就取出来
        Object fileCache = QuickUtils.cache.getSynchronous(yourCacheKey);
        if(fileCache instanceof NewHuiDetailInfo){
            //展示换成数据数据
            onSuccess((NewHuiDetailInfo) fileCache);
        }
    }
    //判断是否没有网络同时缓存中也没有该key的value,提示网络问题
    QuickUtils.net.toast().getNetFail(context,value, "网络不给力");
    //不管如何都走实际调用,getDataFromService(会先判断是否有网,如果就取数据,并更新缓存,没有就不发送请求)
    getDataFromService();
}

//在类似于AsyncTask的onPostExecute()的方法中调用,具体根据你的网络框架来
private void saveCache(NewHuiDetailInfo obj) {
    base_info = obj;
    //有网络才存储,为什么,自己想....
    if(QuickUtils.net.hasNetwork()){
        QuickUtils.cache.setSynchronous(yourCacheKey, obj);
    }
}

ps:你也许会感觉奇怪,为什么要反复的多次都去判断网络,因为有很多代码都是异步的，所以可能你此时没网络，走到下一步的时候就有了网络，为了程序严谨，反复的每次逻辑操作前都附带判断网络.

至于为什么获取到数据后要先判断是否有网络才去存储，自己思考下.相通了你会发觉这里的逻辑很有意思.

四.后记
